> Talks, demos, podcasts, panels: how we choose them, prepare for them, and follow up.

This document describes how Forgejo Society participates in public events.
The canonical event records live in [`events/`](events/README.md). Talk
abstracts and slide outlines themselves are drafted in
[`../FORGEJO-SOCIETY-PROMOTION/talks/`](../promotion/talks/README.md);
this folder records the *appearances* and what came of them.

---

## What counts as an event

For the purposes of publicity, an **event** is any public appearance in
which the project is represented by name. This includes:

- Conference talks, lightning talks, and tutorials.
- Podcast and video interviews.
- Live demos and workshops.
- Panel discussions and roundtables.
- Public AMAs, office hours, and community calls hosted by other
  projects.

Internal calls between maintainers, private demos under NDA, and casual
mentions in someone else's talk are **not** events for this purpose.

## Selection criteria

We say yes to events that meet at least the following:

1. **Audience fit.** The audience plausibly includes maintainers,
   researchers, operators, or critics whose work overlaps with the
   project.
2. **Substance over surface.** The format allows enough time to
   describe mechanisms, not only slogans.
3. **Sustainable load.** The event does not crowd out the project's
   primary work.

We say no, with thanks, to events that require us to overstate the
project, to attack peer projects, or to commit to publishing on a
schedule we cannot meet.

## Preparation

For each accepted event:

1. **Open an event record** in [`events/`](events/) named:

   ```
   event-YYYY-MM-DD-short-handle.md
   ```

2. **Draft the talk artefacts** (abstract, outline, demo script) in
   [`../FORGEJO-SOCIETY-PROMOTION/talks/`](../promotion/talks/README.md)
   and link them from the event record.
3. **Identify the speaker(s)** and the author of record for the event
   entry.
4. **Verify all claims** in the talk against the specification, plan,
   setup, or runtime, exactly as for an announcement.

## Follow-up

Within two weeks of each event, the event record is updated with:

- A link to the recording or transcript, if one exists.
- Slides as published, if applicable.
- A short, honest retrospective: what worked, what did not, and what
  questions the audience asked that the project should address in
  documentation.
- Any coverage that resulted, cross-linked from
  [`coverage/`](coverage/README.md).
