The **`repository_dispatch`** event in GitHub is a way to **trigger workflows (GitHub Actions) from outside a repository**. Think of it as a custom webhook you can fire manually via API to start automation.

---

## � What is `repository_dispatch`?

It’s a **custom event** you send to a GitHub repository using the GitHub API. When received, it can trigger a workflow defined in that repo.

� Unlike `push`, `pull_request`, etc., this event:

* Is **not automatic**
* Is triggered **programmatically**
* Can carry **custom data**

---

## ⚙️ How it works

### 1. Define a workflow that listens for it

```yaml
on:
  repository_dispatch:
    types: [my-custom-event]

jobs:
  run:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Triggered by repository_dispatch!"
```

* `types` lets you define **custom event names**
* Only matching events will trigger the workflow

---

### 2. Trigger it via API

You send a POST request to GitHub:

```bash
curl -X POST https://api.github.com/repos/OWNER/REPO/dispatches \
  -H "Accept: application/vnd.github+json" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -d '{
    "event_type": "my-custom-event",
    "client_payload": {
      "key": "value"
    }
  }'
```

---

## � Key components

### � `event_type`

* Required string
* Matches the `types` in your workflow
* Example: `"deploy"`, `"run-tests"`

### � `client_payload`

* Optional JSON object
* Passed into the workflow
* Access it in your workflow like:

```yaml
${{ github.event.client_payload.key }}
```

---

## � Common use cases

### � Cross-repository automation

Trigger workflows in another repo:

* Repo A finishes → triggers Repo B

### � External systems integration

Trigger workflows from:

* CI/CD tools (e.g. Jenkins)
* Backend services
* Cron jobs on external servers

### � Custom pipelines

Run workflows based on:

* Database events
* User actions
* Third-party webhooks

---

## � Permissions required

To trigger `repository_dispatch`, you need a token with:

* `repo` scope (for private repos)
* Or appropriate fine-grained permissions

---

## ⚠️ Important notes

* It **only triggers workflows on the default branch**
* You must define the workflow **before sending the event**
* Rate limits apply (GitHub API limits)

---

## � When to use it vs alternatives

| Use case             | Better option           |
| -------------------- | ----------------------- |
| Internal repo events | `push`, `pull_request`  |
| Manual trigger       | `workflow_dispatch`     |
| External trigger     | ✅ `repository_dispatch` |

---

## � Simple mental model

* `workflow_dispatch` → manual click in GitHub UI
* `repository_dispatch` → trigger from **code or external system**
