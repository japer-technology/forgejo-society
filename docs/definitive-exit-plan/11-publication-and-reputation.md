# Publication and Reputation Strategy

Public visibility should be treated as a governed publication layer, not as the place where internal work lives.

## Role in the exit plan

- Keep private and active work in Forgejo.
- Reflect selected repositories outward to GitHub and other public surfaces on purpose.
- Preserve public credibility and discoverability without making GitHub the center of operations.

## Repository publication classes

- **Private only**: stays inside Forgejo.
- **Mirrored publicly**: published outward for visibility and backup, but canonical in Forgejo.
- **Open for external contribution**: published outward with an explicit intake and triage policy.
- **Archive**: retained for continuity with clear read-only status.

## Publication rules

- Make publication a release or promotion step rather than an automatic side effect of daily work.
- Decide which metadata is mirrored outward and which stays private.
- Keep repository descriptions, release notes, and public links consistent across mirrors.
- Make the canonical source of truth explicit on every public mirror.

## Public surfaces

- Use GitHub for reputation, discovery, releases, and public proof of work.
- Use Codeberg or GitLab as additional public continuity surfaces where appropriate.
- Keep mirror cadence, ownership, and recovery procedures documented.

## Open decisions

- Which projects are public showcase projects versus private internal work?
- Which public mirrors accept issues or pull requests directly?
- What approval is required before a private Forgejo repository becomes public?
